Kimi Code Web 消息同步逻辑形状

这份图说明 Web UI 如何通过 REST snapshot + WebSocket 增量事件来显示消息，以及为什么会出现短时间多次 resync_required / snapshot 请求风暴。

0. 图例

REST HTTP 请求/响应，例如 GET /snapshot、POST /prompts。

WebSocket 长连接，服务端把事件增量推给前端。

Event Bus agent-core 内部事件总线，所有会话事件都先进入这里。

Client State 前端 composable 里的响应式状态，驱动 UI 渲染。

1. 总体组件

Browser UI Vue 组件、会话列表、聊天面板、输入框。
useKimiWebClient 前端状态中枢：保存 sessions、messages、lastSeq、epoch、WS connection。
Daemon API Client REST client + WebSocket client，负责和本地 server 通信。
↓ HTTP / WS
REST Routes /snapshot、/prompts 等。
WSBroadcastService 把事件总线事件转成 WS frame，并维护 per-session seq / epoch。
SessionEventJournal 每个 session 一个 JSONL journal，记录 durable events 和 seq。

2. 打开一个 session：snapshot 全量同步

selectSession(sessionId)
  └─ syncSessionFromSnapshot(sessionId)
      ├─ GET /api/v1/sessions/{id}/snapshot
      ├─ SnapshotService 读取 SessionStore / state.json / wire.jsonl
      ├─ WSBroadcastService.getSnapshotState() 返回当前 seq / epoch / inFlightTurn
      ├─ 前端写入 messages / approvals / questions / lastSeq / epoch
      ├─ eventConn.seedSnapshot(...) 恢复正在进行的 turn
      └─ eventConn.subscribe({ seq: as_of_seq, epoch })

关键点：snapshot 是“当前水位的全量状态”，WebSocket 负责从这个水位之后开始增量同步。

3. 接收实时消息：WebSocket 增量事件

agent-core 内部产生事件
  └─ IEventService.publish(event)
      └─ WSBroadcastService
          ├─ durable event：分配 seq，写入 journal，加入内存 tail
          ├─ volatile event：不分配新 seq，只带当前 watermark 直接广播
          └─ 对订阅了该 session 的 WS 连接 fan-out

前端收到 WS frame 后：

DaemonEventSocket
  └─ classifyFrame / agentEventProjector / eventReducer
      └─ 更新 rawState.messagesBySession / lastSeqBySession / status
          └─ Vue 组件重新渲染

durable event 推进 lastSeq；volatile delta（如 thinking.delta）用于实时显示，不推进 durable seq。

4. 发送消息：从输入框到事件流

Composer 提交 prompt
  └─ POST /api/v1/sessions/{id}/prompts
      └─ agent-core 开始处理
          └─ 产生 message.created / turn.started / assistant.delta / tool.* / turn.ended 等事件
              └─ 这些事件重新进入 Event Bus → WSBroadcast → WebSocket → 前端渲染

所以“发送消息”的写入路径是 REST；发送之后看到的过程和结果，主要通过 WebSocket 事件流回来。

5. 什么时候会 resync_required？

客户端 subscribe / client_hello 会带上自己的同步游标：

{ seq: lastSeq, epoch: lastEpoch }

服务端在 getBufferedSince() 里判断能不能继续增量回放：

  • epoch 不一致：客户端拿着旧 journal 的 epoch，必须重建。
  • 客户端 seq 比服务端还新：cursor 不可信，强制重建。
  • 落后超过 1000 条 durable events：回放成本太高，改用 snapshot 重建。
subscribe(cursor)
  └─ syncSessions()
      ├─ 先把订阅注册上（live events 可以立刻流过来）
      └─ getBufferedSince(cursor)
          ├─ 可回放：发送漏掉的 durable events
          └─ 不可回放：发送 resync_required

6. 实际运行形状：为什么会短时间多个 resync？

正常应该是：

subscribe(old cursor) → resync_required → snapshot → subscribe(new cursor)

但前端有多条路径都会触发 subscribe，并且可能都使用旧 cursor：

  • 切回 session 时重新订阅；
  • WebSocket 断线重连后重新 client_hello；
  • 路由 popstate / active session 变化；
  • 上一次 snapshot 还没回来时，又有别的路径触发 subscribe。
1. 切回 running session，前端用旧 lastSeq subscribe。
2. 服务端发现落后太多，回 resync_required。
3. 旧前端开始 GET /snapshot。
4. snapshot 还没回来，另一条路径又带着旧 cursor subscribe。
5. 服务端再次回 resync_required。
6. 旧前端再次 GET /snapshot，形成并发/堆积。

所以问题不是“单个 snapshot 慢”，而是多个旧 cursor 订阅触发多个 resync，多个 resync 又触发多个 snapshot。

7. 修复后的形状：coalesced snapshot sync

修复加入了一个按 session 去重的 runner：

createCoalescedAsyncRunner(syncSessionFromSnapshot)

规则：

  • 同一个 session 已有 snapshot 在加载：不重复发起。
  • 加载期间又来 resync：只记录“需要再跑一次”。
  • 当前加载完成后：最多再补跑一次，拿最新状态。
resync_required A → 开始 snapshot #1
resync_required B → snapshot #1 还在跑，只排队一次
resync_required C → 不重复排队
snapshot #1 完成 → 再跑 snapshot #2，拿到最新水位

结果：最多只有 1 个 in-flight snapshot + 1 个 queued rerun，不会再出现请求风暴。

8. 一句话总结

snapshot 是“全量重建”，WebSocket 是“增量同步”。 当客户端 cursor 过期或落后太多时，服务端会发 resync_required 让前端重新 snapshot。 旧前端没有对同 session 的 snapshot 去重，所以短时间多次 resync 会放大成并发 snapshot 请求风暴； 修复后同 session 的 snapshot sync 被串行化，并最多保留一次后续刷新。
