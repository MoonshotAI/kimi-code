Tab / 分屏逻辑整理与成熟版设计方案

结论：当前实现已经收敛成「对话」和「文件」两个持久 tab，Preview 只应作为工作区外资源或媒体的临时面板。下一版建议把分屏视为“工作区布局”，把任务、Todo、BTW、Subagent 详情都放进 dock 或右侧临时面板，不再强行成为每个分屏都携带的主 tab。

先给结论

保留
每个 split group 只持久保留 chat 和 files。这是现在代码最清晰的部分，符合重复使用的主工作流。
收口
preview 不应该常驻 tab。工作区内文件直接进 Files；工作区外文件、媒体、Thinking、Compaction、Subagent detail 走临时右侧详情层。
建议
分屏按钮只复制“当前 group 的工作区视图”，默认新 group 打开另一个主视图：Chat 分屏出 Files，Files 分屏出 Chat。任务类信息不参与 group tab。

真实代码入口

  • 布局状态apps/kimi-web/src/composables/usePaneLayout.ts
  • tab 渲染apps/kimi-web/src/components/TabBar.vue
  • 主体承载apps/kimi-web/src/components/ConversationPane.vue
  • 工作 dockConversationPane.vue 底部 dock-workbar

当前状态

  • 持久 tabALL_VIEWS = ['chat', 'files']，每个 group 默认只有对话和文件。
  • Preview由 openPreview() 临时插入，不写入默认 tab；reload 时会被清掉。
  • Tasks/TodoPaneKey 仍允许 tasks/todo，但当前主 UI 主要通过底部 dock 打开。
  • Split拆分当前 group，另一边默认打开相反主视图：当前 Files 则新 group Chat，否则 Files。

现在奇怪的点

历史痕迹
PaneKey 还包含 tasks/todo，但 TabBar 不展示它们，读代码时会觉得“能作为 tab，但 UI 又没有”。
分屏负担
每个 group 都带完整 tab 条和 split 按钮，视觉上像每块区域都能承载全部能力，但真实只有 Chat/Files 是稳定视图。
Preview 误用
工作区内产物如果还走 Preview，会增加一个临时 tab，用户会在 Files 和 Preview 之间迷路。

视图分层

层级承载内容规则
主 tab对话、文件每个 split group 都固定存在，用户可重复使用。
文件子视图Changed、All、Diff、File Preview属于 Files 内部状态，不再新增全局 Preview tab。
底部 dockComposer、Bash task、Subagent、Todo、Goal跟当前 session 绑定，不跟每个 split group 绑定。
右侧详情Thinking、Compaction、Subagent detail、工作区外 Preview临时、互斥、可关闭，不进入持久布局。

建议的数据模型

{
  "layout": {
    "type": "split",
    "dir": "row",
    "children": [
      { "type": "group", "views": ["chat", "files"], "active": "chat" },
      { "type": "group", "views": ["chat", "files"], "active": "files" }
    ],
    "sizes": [1, 1]
  },
  "filesStateBySession": {
    "sess_1": { "mode": "changed", "selectedPath": "reports/out.html" }
  },
  "detailPanel": { "kind": "subagent", "id": "agent_1" }
}

重点：layout 只描述空间；Files 选择、详情面板、dock 展开状态都从 layout 里拆出去。

交互规则

动作结果原因
点击右上角 git 区域当前 Chat group 切到 Files，并选中 Changed。这是“看改动”的直接路径，不需要用户先找文件 tab。
点击对话中的工作区内文件链接打开 Files tab，左侧定位文件，右侧展示 diff 或内容。文件属于工作区，应该进入文件工作流。
点击工作区外路径或媒体打开右侧临时详情面板。Files API 不能保证能读工作区外资源。
拆分 Chat group新 group 默认打开 Files。最常见的并排需求是“对话 + 文件”。
拆分 Files group新 group 默认打开 Chat。避免两边都显示同一个 Files 初始态。
打开 Task/Subagent/Todo底部 dock 展开，必要时右侧详情展示完整输出。任务是 session 状态，不是每个 split 的独立 tab。

状态流

落地步骤

清理类型边界：把 PaneKey 拆成 PrimaryPaneKey = 'chat' | 'files' 和 TransientPaneKey = 'preview'。Tasks/Todo 不再进入 pane layout。
沉淀 Files 状态：为每个 session 保存 mode、selectedPath、selectedLine，从 ConversationPane 的局部状态逐步提到 composable。
统一打开入口：保留一个 openWorkspaceFileInFiles()，所有工作区内文件链接、git 区域、改动列表都走这一条。
收口详情层：Thinking、Compaction、Subagent detail、工作区外 Preview 共用一个 detailPanel 状态，互斥展示。
最后清 UI：TabBar 只展示对话/文件；Preview 不常驻；split 操作留在 group 右上角，但只影响空间布局。

验收标准

通过新开 session 只看到对话和文件两个主 tab。
通过工作区内 HTML/Markdown/代码产物链接打开到 Files，不新增 Preview tab。
通过工作区外路径仍有可见反馈，不会静默失败。
通过拆分后两边默认互补显示 Chat 和 Files。
通过刷新页面后不会恢复空 Preview，也不会丢失 Chat/Files split。
