调研存档：「手动终止」提示能否进入信息流

2026-06-11 · 仓库 kimi-code-web @ feat/web · 由 4 个并行核查代理对原调研报告逐条验证（事件链 / 持久化 / UI 现状 / 方案可行性），所有 file:line 均已核实

1. 核心结论

① daemon 存了，但取不回。abort 产生的 turn.ended(reason='cancelled') 和 prompt.aborted 都会持久化进事件日志（<kimiHome>/daemon/events/<sid>.jsonl），但该日志只服务于断线重连补发；刷新后走 snapshot 加载，没有任何 API 能拿到历史 turn 的结束原因。

② 实时路径上，取消信号只活到 session 状态层。到消息流（AppMessage → ChatTurn）的转换中信息全部丢失——这部分原报告诊断正确。

③ 原报告的两条改造方案都有事实错误：注入点选错了层（eventReducer 收不到 turn.ended），且最常见的中途打断路径上 messageUpdated 根本不会发出（见 时序陷阱）。

2. 原始问题与报告判定总览

原始问题：能否把"手动终止"的提示直接显示在信息流（chat turns）中而不是 toast？daemon 是否会存储这个中断事件？

2.2 原报告错了 / 漏了的

论断判定实际情况
（未回答）daemon 是否存储中断事件 关键遗漏 journal 持久化了，但无 API 取回（见第 4 节）
方案 B：「eventReducer 收到 turn.ended 时注入 metadata」 事实错误 AppEvent 联合类型没有 turn.* 档位，turn.ended 在 projector 层已被消化成 messageUpdated / sessionStatusChanged / sessionUsageUpdated。注入点只能在 agentEventProjector
方案 A：「给 messageUpdated 的 status 加 'cancelled'」 主路径空转 中途打断时 turn.ended 分支拿不到消息 id，messageUpdated 根本不发（见第 6 节时序陷阱）
「目前用 toast 的方案（已实现）」 只对一半 toast「您已手动终止」只在 Escape 键路径弹出（ConversationPane.vue:478-498，commit fca2abe4）；点停止按钮终止时没有任何提示；且 toast 是按键时乐观弹出，不等 daemon 确认
「reason==='cancelled' 时 session status 推为 'aborted'」 不完整 reason='failed' 同样映射为 'aborted'（agentEventProjector.ts:657）；且 'aborted' 没有任何 UI 组件消费——toUiSessionStatus 把它折叠成 'idle'（useKimiWebClient.ts:694-699）

4. 持久化：存了但取不回

数据源abort 留下了什么刷新后能否取回
事件日志 journal
<kimiHome>/daemon/events/<sid>.jsonl		 turn.ended(cancelled)、prompt.aborted 都是 durable 事件，确实落盘（wsBroadcastService.ts:104-111） 否 重放仅限同生命周期断线重连（旧 cursor + epoch 匹配 + gap ≤ 1000）；刷新后订阅 cursor = snapshot 的 as_of_seq，历史事件永不重放
消息持久化
services/message/message.ts:259-272		 什么都没存。metadata 只可能带 origin；message schema 无 finish_reason（protocol/src/message.ts:73-82） 否
snapshot
protocol/src/rest/snapshot.ts:60-75		 session.status 写死 'idle'（services/session/session.ts:105）；in_flight_turn 在 turn.ended 时即清空（inFlightTurnTracker.ts:64-65） 否
agent-core records 有一条 turn.cancel 持久记录（agent/turn/index.ts:206），但 resume 语义仅恢复状态机，不进客户端可见的回放；turn.interrupted 是 LiveOnly 事件不入 transcript，被中断 step 连 step.end/content.part 都不写——已流出的部分文本也会丢 否
唯一间接痕迹 abort 时若有工具在执行，落一条 is_error 的 tool_result，文本 "The user manually interrupted …"（loop/tool-call.ts:56-60） 可见但非结构化 纯文本生成阶段被打断则毫无痕迹

TUI 同样没有重放标记：AgentReplayRecord 只有 6 种变体，无 interrupted/cancelled；"Interrupted by user" 只在 live 事件时显示。唯一例外是 goal 驱动的 turn（goal_updated 重放可显示 "Goal paused due to user's interruption"）。

5. 当前 UI 现状（2026-06-11）

  • Esc 终止：乐观弹 3 秒 toast「您已手动终止」（ConversationPane.vue:478-498, :784-794；i18n conversation.manuallyAborted）。
  • 点停止按钮终止：emit('interrupt') 直接转发（Composer.vue:740 → ConversationPane.vue:588/772），无任何提示。
  • abort 失败：另出一条警告 toast（useKimiWebClient.ts:2018 → WarningToasts）。
  • 'aborted' 状态：被 toUiSessionStatus 折叠为 'idle'，无组件消费；abort 还会触发 onSessionIdle 的副作用——自动 flush 该会话排队中的下一条 prompt（useKimiWebClient.ts:1299-1315）。

6. 时序陷阱（改造前必须知道）

手动 abort 落在步骤中途时（最常见场景），事件顺序是：turn.step.interrupted 先到，projector 在该分支把 s.currentAssistantMsgId 清空（agentEventProjector.ts:678-683）；随后 turn.ended 分支 if (msgId) 为假——被中止的 assistant 消息不会收到任何 messageUpdated。即使 abort 落在两步之间，tool.result 也已把 msgId 清空（:586）。

因此「给 messageUpdated 加 cancelled status」或「在 turn.ended 时注入 metadata」在主路径上都拿不到目标消息。正确做法：在 projector 处理 turn.step.interrupted（带 reason==='aborted'，可区分手动中断 / max_steps / error）时、清空 msgId 之前记录或打标。

7. 改造方案（修正版）

方案一：仅实时显示（纯前端）方案二：刷新后仍可见（动 daemon）
效果 本次会话内，被终止的 turn 末尾渲染「已手动终止」行；刷新后消失 终止标记进入持久数据，任何时候加载都显示
注入点 projector 的 turn.step.interrupted(reason='aborted') + turn.ended(cancelled) 两处（不是 eventReducer） daemon/services 把结束原因写进消息 metadata（如 finish_reason），或 snapshot 暴露 journal 中已有的 turn.ended；数据源现成
涉及文件 agentEventProjector.ts、types.ts（AppEvent/AppMessage）、eventReducer.ts、src/types.ts（ChatTurn）、messagesToTurns.ts、ChatPane.vue（桌面 + 移动两套模板）；删 toast 动 ConversationPane.vue 左列全部 + protocol message schema、services/message/message.ts、agent-core history 写入点
额外收益 挂到状态事件而非按键上，可统一 Esc 和停止按钮两个入口的提示 同左，且 TUI 将来也可复用同一标记
边界 「空转 turn」：取消时尚无任何 assistant 输出，turn 级标记无处可挂，需考虑渲染独立的系统行
